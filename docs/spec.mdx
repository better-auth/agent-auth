# Agent Auth Protocol

**Version:** 0.1 (Draft)

## 1. Introduction

### 1.1 Problem Statement

AI agents increasingly act on behalf of users — reading emails, creating issues, managing calendars. The current approach is to share the user's OAuth token or API key with the agent. This creates three problems:

1. **No per-agent identity.** The server can't distinguish between agents sharing the same OAuth client, or between an agent and the user themselves — they all present the same token.
2. **Over-privileged access.** The shared token has all the user's permissions. An agent that only needs to read emails gets write access to everything.
3. **No per-agent control.** You can't revoke one agent's access without invalidating the token for all agents sharing the same client.

### 1.2 Solution

Agent Auth gives each AI agent its own cryptographic identity (Ed25519 keypair) and manages permissions separately from the user's session. Agents authenticate via signed JWTs, not shared tokens. Permissions are granted per-agent, per-scope, per-resource, by individual users.

The fundamental guarantee: **the server always knows exactly which agent performed an action, what permissions it had, and who authorized each permission.** Actions are never attributed to a host or client — they're attributed to the specific agent that made the request.

### 1.3 End-to-End Flow

1. host_jwt (signed by host's private key)
2. host_public_key (Ed25519 public key) -> 
2. agent_public_key
3. scopes_requested

```
┌─────────────────────────────────────────────────────────────────────┐
│  1. DISCOVERY                                                       │
│                                                                     │
│  User: "Send email via Gmail"                                       │
│       ↓                                                             │
│  Agent fetches gmail.com/.well-known/agent-configuration            │
│       ↓                                                             │
│  ┌─ Scopes public? ────────────────────────────────────────────┐    │
│  │                                                              │    │
│  │  YES → agent gets: issuer, endpoints, algorithms, scopes    │    │
│  │         agent selects scopes → skip to REGISTRATION          │    │
│  │                                                              │    │
│  │  NO  → agent gets: issuer, endpoints, algorithms (no scopes)│    │
│  │         agent asks host to discover scopes (see below)       │    │
│  └──────────────────────────────────────────────────────────────┘    │
└─────────────────────────┬───────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────────────┐

│                                                                     │

│       ↓                                                             │
│  Host signs JWT → sends to auth server's config endpoint            │
│       ↓                                                             │
│  ┌─ Host known? ───────────────────────────────────────────────┐    │
│  │  YES → server returns scopes for this user                   │    │
│  │  NO  → server triggers device auth → user approves host      │    │
│  │         → server returns scopes                              │    │
│  └──────────────────────────────────────────────────────────────┘    │
│       ↓                                                             │
│  Host returns scopes to agent → agent selects what it needs         │
└─────────────────────────┬───────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────────────┐
│  3. REGISTRATION                                                    │
│                                                                     │
│  Agent generates a keypair for this service                         │
│       ↓                                                             │
│  Agent → Host: "register me with these scopes"                      │
│       ↓                                                             │
│  Host → Auth Server: POST /agent/create + host_jwt + agent_key      │
│       ↓                                                             │
│  ┌─ Host known? ───────────────────────────────────────────────┐    │
│  │  YES → ┌─ Scopes in host's pre-auth? ─────────────────┐     │    │
│  │        │  ALL  → agent created, permissions active      │     │    │
│  │        │  SOME → matched active, rest pending           │     │    │
│  │        │  NONE → all pending → user approves            │     │    │
│  │        └────────────────────────────────────────────────┘     │    │
│  │  NO  → device auth → user approves host + agent               │    │
│  │        user optionally trusts host with default scopes        │    │
│  └──────────────────────────────────────────────────────────────┘    │
│       ↓                                                             │
│  Agent receives agent_id                                            │
└─────────────────────────┬───────────────────────────────────────────┘
                          ↓

│  4. AUTHENTICATED REQUESTS                                          │
│                                                                     │
│  Agent signs JWT with its private key (sub = agent_id)              │
│       ↓                                                             │
│  Agent → Resource Server: request + Authorization: Bearer <jwt>     │
│       ↓                                                             │
│  Auth Server verifies JWT → resolves permissions → builds session   │
│       ↓                                                             │
│  Resource Server checks permissions                                 │
│       ↓                                                             │
│  YES → process request    NO → 403 (agent can escalate scopes)     │
└─────────────────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────────────────┐
│  5. SCOPE ESCALATION (if needed mid-task)                           │
│                                                                     │
│  Agent needs more permissions than it started with                  │
│       ↓                                                             │
│  Agent → Auth Server: request-scope + reason                        │
│       ↓                                                             │
│  In host's pre-auth? YES → active immediately                       │
│                      NO  → pending → user approves                  │
│       ↓                                                             │
│  Agent continues with new permissions                               │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.4 Design Principles

- **Every action traces to an agent.** The server never sees "a host did this" — it sees "agent X, registered through host Y, with permissions granted by users A and B, made this request." The host registers agents; the agent authenticates directly.
- **Agents are not users or clients.** An agent acts for a user, but has its own identity and constrained permissions.
- **Proof-of-possession over bearer tokens.** Each agent signs its own requests with its own private key. No replayable secrets traverse the wire.
- **Multi-grantor support.** An agent can accumulate permissions from multiple users, each independently revocable.
- **Host as bootstrap, not intermediary.** Every agent is created through a registered host, providing a verifiable chain of trust. But agents authenticate directly — the host's keypair is only for registration, not for ongoing requests.

## 2. Architecture

### 2.1 Actors

The protocol defines three actors:

| Actor | Role | Example |
|---|---|---|
| **Agent** | The AI identity that makes authenticated requests | An LLM completing a task |
| **Host** | A registered credential with pre-authorized scopes that agents are created through | A device, a hosted service (ChatGPT, Claude), a network, an org's private server, an MCP server, a CLI tool |
| **Authorization Server** | Manages hosts, agents, and permissions. Verifies JWTs, handles registration and approval flows. | Better Auth with the agent-auth plugin |
| **Resource Server** | Hosts the protected APIs that agents call. Checks agent permissions before allowing access. | The app's API endpoints |

The authorization server and resource server MAY be the same deployment (common for monolithic apps) or separate services (common for microservices). The protocol does not prescribe the deployment model.

### 2.2 Communication Flow

**Registration:** Host authenticates with its own key → creates the agent on the authorization server

**Request time:** Agent signs its own JWT → sends directly to the resource server

Each agent has its own unique keypair. The agent's private key MUST be stored per-agent — it is not shared with other agents. How the agent accesses its private key (tool call, native crypto, SDK, CLI) is an implementation detail. The host's keypair is only used for bootstrapping (registering new agents) and MUST NOT be used for agent authentication.

The JWT's `sub` claim is the agent ID, and the authorization server verifies against the agent's stored public key. The server always knows exactly which agent made the request.

### 2.3 Consent Flow

When an agent needs permissions that aren't pre-authorized by the host, the protocol uses the device authorization flow ([RFC 8628](https://datatracker.ietf.org/doc/html/rfc8628)) as its standard consent interface:

1. The server returns a `verification_uri` to the host
2. The host presents the URL to the user (opens a browser, displays a code, etc.)
3. The user approves at that URL
4. The host polls until approval completes

What happens behind the `verification_uri` is the server's concern — it could render an OAuth consent page, a custom approval UI, trigger a push notification, or anything else. The host sees a standardized interface: a URL and a polling mechanism. This means host implementations are portable across servers regardless of how each server implements its approval experience.

For pre-authorized scopes (within the host's scope budget), no consent flow is needed — the agent is created immediately.

### 2.4 Trust Model

- The **authorization server** trusts hosts it has registered (verified by public key)
- The **host** registers agents on the authorization server using its own keypair. It does not proxy ongoing requests.
- **Users** grant permissions to agents — either pre-authorized via the host's scope budget, or explicitly approved per request
- The **agent** authenticates directly with the resource server using its own keypair
- The **resource server** delegates JWT verification and permission resolution to the authorization server
- Every request is attributable to a **specific agent**, never to a host or client

## 3. Entities

### 3.1 Agent Host

A host is a registered client runtime. It has:

| Field | Type | Description |
|---|---|---|
| `id` | string | Unique identifier |
| `user_id` | string (FK → user) | The user who registered this host |
| `scopes` | string[] | Pre-authorized scope budget — agents created through this host can be auto-approved for these scopes |
| `public_key` | string? | Ed25519 public key as JWK JSON (required if no `jwks_url`) |
| `jwks_url` | string? | JWKS URL for remote key discovery (required if no `public_key`) |
| `status` | enum | `"pending"`, `"active"`, `"expired"`, or `"revoked"` |
| `activated_at` | datetime? | Last activation timestamp |
| `expires_at` | datetime? | Current TTL deadline |
| `last_used_at` | datetime? | Last time a host JWT was used |
| `created_at` | datetime | Creation timestamp |
| `updated_at` | datetime | Last modification timestamp |

**Invariants:**
- A host MUST have either a `public_key` or a `jwks_url` (at least one is required)
- A host is always owned by exactly one user
- The `scopes` array defines the maximum set of permissions that can be auto-approved for agents created through this host
- Hosts follow the same three-state lifecycle as agents (Section 5)

### 3.2 Agent

An agent is a cryptographic identity. It has:

| Field | Type | Description |
|---|---|---|
| `id` | string | Unique identifier |
| `user_id` | string (FK → user) | The user this agent acts for |
| `host_id` | string (FK → host) | The host that created this agent (**required**) |
| `status` | enum | `"pending"`, `"active"`, `"expired"`, or `"revoked"` |
| `public_key` | string | Ed25519 public key as JWK JSON |
| `last_used_at` | datetime? | Last authenticated request |
| `activated_at` | datetime? | Last activation timestamp |
| `expires_at` | datetime? | Current TTL deadline |
| `metadata` | object? | Arbitrary key-value data |
| `created_at` | datetime | Creation timestamp |
| `updated_at` | datetime | Last modification timestamp |

**Invariants:**
- Every agent MUST be created through a registered host (`host_id` is required)
- An agent MUST have a valid Ed25519 public key
- The agent table contains NO authorization data — permissions live in the permission table
- Agents follow the three-state lifecycle (Section 5)

### 3.3 Agent Permission

A permission is a single authorization grant for an agent. An agent's effective permissions are the union of all its active permission rows.

| Field | Type | Description |
|---|---|---|
| `id` | string | Unique identifier |
| `agent_id` | string (FK → agent) | The agent this permission belongs to |
| `scope` | string | The permission string (opaque to the protocol) |
| `reference_id` | string? | Resource this permission applies to (null = unrestricted) |
| `granted_by` | string (FK → user) | The user who granted this permission |
| `status` | enum | `"active"`, `"pending"`, or `"denied"` |
| `reason` | string? | Human-readable reason (for pending requests) |
| `expires_at` | datetime? | Per-permission expiry |
| `created_at` | datetime | Creation timestamp |
| `updated_at` | datetime | Last modification timestamp |

**Invariants:**
- Only permissions with `status = "active"` are enforced
- Permissions with `status = "pending"` represent unapproved scope requests
- The `scope` field is opaque — the protocol stores and matches it, the application defines the format
- The `reference_id` enables per-resource scoping (e.g. "email.read" on user A vs user B)
- The `granted_by` field enables multi-grantor: different users can grant different permissions to the same agent
- Deleting an agent MUST cascade-delete all its permissions

## 4. Authentication

### 4.1 Keypair Generation

Both hosts and agents use Ed25519 keypairs. The protocol uses JWK (JSON Web Key) format.

**Requirements:**
- The key algorithm MUST be Ed25519 (curve `"Ed25519"`, algorithm `"EdDSA"`)
- The private key MUST never leave the client (host or agent runtime)
- Each keypair SHOULD include a unique `kid` (key ID) for identification
- Servers MAY support additional algorithms via configuration

### 4.2 JWT Signing

Agents authenticate by signing short-lived JWTs with their private key.

**Required claims:**

| Claim | Description |
|---|---|
| `sub` | The agent ID |
| `iat` | Issued-at timestamp (Unix seconds) |
| `exp` | Expiration timestamp |
| `jti` | Unique token identifier (for replay detection) |

**Required headers:**

| Header | Description |
|---|---|
| `alg` | `"EdDSA"` |
| `typ` | `"JWT"` |
| `kid` | Key ID (if available) |

**Token lifetime:** Servers SHOULD enforce a maximum JWT age (default: 60 seconds). Short-lived tokens limit the window for misuse.

### 4.3 Request Binding (Optional)

JWTs MAY include DPoP-style claims that bind the token to a specific HTTP request, following conventions from [RFC 9449](https://datatracker.ietf.org/doc/html/rfc9449):

| Claim | Description |
|---|---|
| `htm` | HTTP method (e.g. `"POST"`) |
| `htu` | Request path (e.g. `"/api/emails/send"`) |
| `ath` | SHA-256 base64url hash of the request body |

When present, the server MUST verify these claims against the actual request. A mismatch MUST result in rejection. When absent, the server MUST accept the JWT without request binding verification.

This means a stolen JWT with request binding can only authorize the exact request it was signed for.

### 4.4 JWT Verification

The server verifies agent JWTs as follows:

1. Decode the JWT and extract the `sub` claim (agent ID)
2. Look up the agent by ID — this identifies the **specific agent**
3. Check agent status (revoked → reject, expired → attempt reactivation)
4. Parse the agent's stored public key
5. Verify the JWT signature against the public key
6. Check `exp` and `iat` against the server's `jwt_max_age`
7. Check `jti` against the replay cache (Section 4.5)
8. If `htm`/`htu` claims are present, verify against the actual request (Section 4.3)
9. Resolve the agent's active permissions from the permission table
10. Build the agent session — this contains the agent's identity, its specific permissions, who granted each one, and which host created it

The resulting session gives the server complete attribution: **which agent** made the request, **what it's allowed to do**, **who authorized each permission**, and **which host created it**.

### 4.5 JTI Replay Detection

All JWTs MUST include a `jti` (JWT ID) claim. The server maintains an in-memory cache of seen JTI values. If a JWT's `jti` has been seen within its `jwt_max_age` window, the request MUST be rejected.

This prevents:
- Replay attacks (intercepted JWTs being reused)
- Double-processing of requests

## 5. Lifecycle

### 5.1 Three-State Model

Both hosts and agents follow a three-state lifecycle:

```
                   user re-approves
      ┌──────────────────────────────────┐
      │                                  │
      ▼          TTL / max_lifetime      │
  ┌────────┐  ─────────────────────►  ┌──┴────────┐
  │ active │                          │  expired   │
  └────────┘                          └───────────┘
      │                                    │
      │   absolute_lifetime / revoke       │  absolute_lifetime / revoke
      ▼                                    ▼
  ┌─────────┐                         ┌─────────┐
  │ revoked │                         │ revoked │
  └─────────┘                         └─────────┘
```

- **Active** — operational; each request extends the sliding TTL
- **Expired** — TTL or max lifetime elapsed. The agent is rejected until the user re-approves. This is a security boundary — after inactivity, the agent must be explicitly re-authorized. Existing permissions are preserved but not enforced until reactivation.
- **Revoked** — permanent. The agent's status is set to revoked. It cannot be reactivated; a new agent must be created.

### 5.2 Lifetime Clocks

Two independent clocks govern entity lifetimes:

| Clock | Measured From | Resets On Reactivation | Effect |
|---|---|---|---|
| `agent_session_ttl` | last request (`expires_at`) | Yes | → expired |
| `absolute_lifetime` | `created_at` | Never | → revoked (permanent) |

### 5.3 Reactivation

When an expired agent attempts a request, the server rejects with `AGENT_EXPIRED`. To reactivate:

1. **If the agent's permissions are within the host's pre-authorized scopes:** The host can reactivate silently by re-creating the agent with its `host_jwt`. No user interaction needed.
2. **If the agent has permissions beyond the host's scopes:** The host initiates the device authorization flow (Section 2.3). The user must re-approve.

In both cases, existing permissions remain intact — no scope decay, no re-granting needed. The TTL resets on reactivation.

### 5.4 Revocation

Revocation is permanent. When an agent or host is revoked:
- Status is set to `"revoked"`
- For hosts: all agents created under the host are also revoked (cascade)

## 6. Discovery

Before an agent can request permissions or make authenticated calls, it needs to discover what a service supports and where its endpoints are.

### 6.1 Agent Configuration Endpoint

Services implementing Agent Auth MUST expose a configuration document at:

```
GET /.well-known/agent-configuration
```

This follows the same convention as [OpenID Connect Discovery](https://openid.net/specs/openid-connect-discovery-1_0.html) (`.well-known/openid-configuration`).

The response is a JSON document:

```json
{
  "issuer": "https://auth.example.com",
  "scopes": [
    { "name": "email.read", "description": "Read emails" },
    { "name": "email.send", "description": "Send emails" },
    { "name": "contacts.read", "description": "Read contacts" }
  ],
  "algorithms": ["Ed25519"],
  "request_binding": "optional",
  "session_ttl": 3600,
  "endpoints": {
    "host_registration": "/agent/host/create",
    "host_update": "/agent/host/update",
    "agent_creation": "/agent/create",
    "agent_revocation": "/agent/revoke",
    "scope_escalation": "/agent/request-scope",
    "scope_approval": "/agent/approve-scope",
    "agent_session": "/agent/get-session",
    "device_authorization": "/device/code",
    "device_token": "/device/token",
    "key_rotation": "/agent/rotate-key"
  }
}
```

**Always public (no auth required):**
- `issuer` — the authorization server's base URL. May differ from the service URL (e.g. `gmail.com` serves email but `auth.google.com` handles auth).
- `algorithms` — supported key algorithms.
- `endpoints` — where to call for each operation. All paths are relative to the `issuer`.
- `request_binding` — `"required"`, `"optional"`, or `"none"`. Default: `"optional"`.
- `session_ttl` — default TTL in seconds for agent sessions.

**Returned only with authentication:**
- `scopes` — available scopes with human-readable descriptions. Servers MAY require authentication (host JWT or user session) to return scopes, since the available capabilities could be sensitive. When authentication is provided, the server returns scopes relevant to the authenticated user.

This means an unauthenticated request to `.well-known/agent-configuration` returns enough to register (endpoints, algorithms) but not what to request (scopes). The agent registers a host first, then discovers scopes using the host's credentials.

### 6.2 Discovery Flow

**Path A: Scopes are public**

1. Agent fetches `https://service.example.com/.well-known/agent-configuration`
2. Response includes scopes — agent reads them, selects what it needs
3. Agent tells the host: "register me on this service with these scopes" (passes issuer, endpoints, selected scopes, agent public key)
4. Host handles registration (see Section 10)

**Path B: Scopes require authentication**

1. Agent fetches `https://service.example.com/.well-known/agent-configuration`
2. Response has no scopes (auth required) — but has issuer, endpoints, algorithms
3. Agent tells the host: "discover scopes on this service for me"
4. Host signs a JWT with its keypair, sends it to the config endpoint
5. If the host is known → server returns scopes for this user
6. If the host is unknown → server triggers device auth → user approves the host → server returns scopes
7. Host returns scopes to the agent → agent selects what it needs
8. Agent tells the host: "now register me with these scopes"
9. Host handles registration (see Section 10)

In both paths, the host identifies itself by its keypair — the server verifies the JWT signature against the registered public key (or fetches from JWKS URL). The host doesn't need to store a per-service ID; its keypair is its identity everywhere.

### 6.3 Assisted Discovery

Not every agent has access to web search or HTTP fetching. Discovery can be assisted by tools:

- **MCP tools** — a `discover_service` tool that fetches `.well-known/agent-configuration` for a given URL and returns the parsed result
- **Agent skills** — pre-configured knowledge about known services ("gmail.com uses auth.google.com, here are the scopes")
- **Service registries** — an identity provider or directory that maps service names to their agent-configuration URLs

The protocol does not prescribe how the agent discovers the service URL itself. It only requires that the service publishes a `.well-known/agent-configuration` document and that the agent (or an assisting tool) can fetch it.

### 6.4 Discovery Without `.well-known`

If the agent already knows the authorization server and endpoint URLs (e.g. from a pre-configured skill or hardcoded integration), discovery is not required. The `.well-known/agent-configuration` endpoint is for dynamic, portable discovery — not a mandatory step.

## 7. Making Authorized Requests

Once an agent is registered and has permissions, it makes requests to the server's API:

### 7.1 Request Flow

1. **Agent signs a JWT** with its private key. The JWT contains `sub` (agent ID), `jti` (unique nonce), `iat`, `exp`, and optionally request binding claims (`htm`, `htu`, `ath`).
2. **Agent sends the request** to the resource server with the JWT in the `Authorization: Bearer <jwt>` header.
3. **Resource server receives the request** and detects the JWT (three-part token in the Bearer header).
4. **Authorization server verifies** (see Section 4.4): extracts the agent ID, looks up the agent, verifies the signature against the stored public key, checks expiry, checks JTI replay, verifies request binding if present. (If the authorization server and resource server are co-located, this happens in the same process.)
5. **Authorization server resolves permissions** by querying active `agent_permission` rows for this agent.
6. **Authorization server builds the agent session** — the agent's identity plus its effective permissions — and attaches it to the request context.
7. **Resource server checks permissions.** The route handler inspects the agent session to decide if the agent is authorized for this specific action. This check is application-defined — the protocol provides the permissions, the resource server enforces them.

### 7.2 Permission Enforcement

The authorization server stores and delivers permissions. It does NOT enforce them — that's the resource server's job. The authorization server provides the agent session containing the agent's active permissions. The resource server's route handlers decide:

- "Does this agent have `email.read`?" → check the permissions array
- "Does this agent have `email.read` for this specific user?" → check permissions with matching `reference_id`
- "Is this agent allowed to perform this action?" → whatever logic the resource server needs

This keeps the protocol generic. It works with any authorization model — simple scope checks, RBAC, ABAC, policy engines — because it provides the data, not the decision.

### 7.3 What the Server Knows Per Request

For every authorized request, the authorization server provides:

- **Which agent** is making the request (agent ID, public key verification)
- **What it's allowed to do** (active permissions with scopes, reference IDs, grantors)
- **Who approved each permission** (`granted_by` on each permission row)
- **Which host registered it** (`host_id` on the agent)
- **Which user it acts for** (`user_id` on the agent)
- **What specific request it's making** (if request binding is used: method, path, body hash)

## 8. Permissions

### 8.1 Permission Model

Permissions are stored separately from agent identity. Each permission row represents one scope grant from one user.

An agent's effective permissions at any moment are:

```
SELECT * FROM agent_permission
WHERE agent_id = :agent_id
  AND status = 'active'
  AND (expires_at IS NULL OR expires_at > NOW())
```

### 8.2 Scope Format

The `scope` field is opaque to the protocol. The application defines the format:

- Simple strings: `"email.read"`, `"calendar.write"`
- Hierarchical: `"github.issues.create"`
- With resource qualifiers: `"email.read:user123"`

The protocol stores, matches, and returns scope strings without parsing them.

### 8.3 Reference ID

The `reference_id` field on permissions is an optional identifier for grouping or filtering permissions. It has no authorization semantics in the protocol — the server does not use it for access control decisions. It's purely a convenience for applications to associate permissions with an external entity (an organization, workspace, project, user, etc.).

| scope | reference_id | granted_by | Use case |
|---|---|---|---|
| `email.read` | `org_123` | `bekacru` | Permission associated with org_123 |
| `issues.write` | `workspace_456` | `alice` | Permission associated with a workspace |
| `reports.read` | null | `bob` | No reference — standalone permission |

Applications can query permissions by `reference_id` to answer questions like "what permissions does this agent have for this organization?" but the protocol itself treats it as opaque metadata.

### 8.4 Multi-Grantor Authorization

Different users can grant different permissions to the same agent:

1. Bekacru grants `email.read` scoped to his account
2. Alice grants `calendar.read` scoped to her calendar
3. An admin grants `audit.read` globally

Each grant is independently revocable. Revoking Bekacru's grant does not affect Alice's.

### 8.5 Scope Escalation

An agent is created with an initial set of permissions (from the host's pre-authorized scopes or user approval). But as the agent works on a task, it may discover it needs permissions it wasn't initially granted. Scope escalation lets the agent request additional permissions without being re-created.

For example: an agent created with `email.read` starts drafting a response and realizes it needs `email.send`. Instead of the user creating a new agent, the existing agent requests the additional scope.

The flow:

1. Agent sends a scope escalation request with the desired scopes and an optional reason (e.g. "Need send access to reply to the email")
2. Server checks the requested scopes against the host's pre-authorized scopes:
   - **Within host's scopes:** permissions are created with `status = "active"` immediately. No user interaction needed.
   - **Beyond host's scopes:** permissions are created with `status = "pending"`. The user is notified (device auth URL, push notification, dashboard, etc.) and must approve or deny.
3. The agent's next request includes the new permissions

The agent keeps its identity, its existing permissions, and its keypair. Only the permissions change. This means an agent can grow its capabilities over time as the task demands, without being re-created.

### 8.6 Host Pre-Authorization

Hosts carry a `scopes` array — the pre-authorized scope budget. When an agent is created through a host:

1. The server checks if the requested scopes are a subset of the host's `scopes`
2. If yes: permissions are created with `status = "active"` (auto-approved)
3. If no: the server MAY escalate to user approval, or reject the request

Host scopes are server-side policy — the host itself doesn't know or control what its pre-authorized scopes are. The user or server admin can update a host's scopes at any time without the host's involvement. This means:

- A user can expand a host's scopes after building trust ("add `email.send` to my Cursor host")
- A user can restrict a host's scopes if concerned ("remove `billing.write` from this host")
- The server can programmatically adjust scopes based on policy

## 9. Host Registration

### 9.1 Dynamic Host Registration

Hosts can be registered dynamically as part of the agent creation flow. The first time an unknown host tries to create an agent:

1. Host sends an agent creation request, including its public key
2. Server checks: is this host already registered?
   - **Known host** → proceed with normal agent creation (Section 10)
   - **Unknown host** → trigger device authorization flow for user approval
3. User sees: "A new host wants to create agents on your behalf. Approve?"
4. User approves → server registers the host AND creates the agent in one flow
5. On subsequent requests, the host is recognized and can auto-approve agents within its pre-authorized scopes

This eliminates the need for a separate host registration step. The user never has to think about "registering a host" — they just approve or deny when an agent needs access.

### 9.2 Explicit Host Registration

Hosts MAY also be registered explicitly via a dedicated endpoint. This is useful when:
- An admin wants to pre-register hosts before they're used
- The host needs pre-authorized scopes set up in advance
- The deployment requires host approval before any agent creation is possible

The user provides:
- An Ed25519 public key (the host retains the private key), OR a JWKS URL for remote key discovery
- Optional pre-authorized scopes

### 9.3 Remote JWKS for Hosts

Hosts that are services (rather than local tools) MAY register with a JWKS URL ([RFC 7517](https://datatracker.ietf.org/doc/html/rfc7517)) instead of a static public key:

```
{
    jwks_url: "https://my-host.example.com/.well-known/jwks.json",
    scopes: ["email.read"]
}
```

When a `jwks_url` is registered, the server fetches and caches the host's public keys from that URL. JWT verification matches by `kid` header against the keys in the JWKS.

If no `jwks_url` is provided, the server uses the static `public_key` (the default for local hosts like Cursor or CLI tools).

The server SHOULD cache JWKS responses and refresh them periodically (RECOMMENDED: every 5 minutes or on `kid` miss).

### 9.4 Host as Trust Anchor

Every agent MUST be created through a registered host. This ensures:
- The server has seen and approved the client runtime
- Unknown hosts trigger user approval (and optionally 2FA)
- Every agent traces back to a verifiable client

### 9.5 Idempotent Host Creation

If a host registration request includes the host's `id` from a previous registration, the server SHOULD reactivate and update the existing host instead of creating a duplicate. The host is responsible for persisting its own ID between sessions.

## 10. Agent Creation

### 10.1 First-Time Flow (Unknown Host)

When an agent needs to be created through a host the server hasn't seen before, the full flow is:

1. Host sends agent creation request with its public key and the agent's public key
2. Server detects unknown host → initiates device authorization flow
3. User sees an approval screen where they can:
   - **Approve this agent** — the agent is created with the requested permissions.
   - **Trust this host** (opt-in) — the user explicitly opts in to trusting the host for future agents. The user selects which scopes to pre-authorize for the host. These become the host's default scopes — future agents through this host are auto-approved for those scopes without another approval screen.
4. If the user only approves the agent (without trusting the host), the host is registered with zero pre-authorized scopes. The next agent through this host will require approval again.
5. Host receives the agent ID and begins authenticating

How the server handles host trust is implementation-defined. Some options:

- **Auto-trust on first approval** — approving any agent through a host automatically trusts the host with the approved scopes. Simplest UX.
- **Explicit opt-in** — the approval screen lets the user choose whether to also trust the host, and which scopes to pre-authorize. More control.
- **Always untrusted** — every agent always requires approval regardless. Most restrictive.

The server MAY require additional verification for new hosts (e.g. 2FA).

### 10.2 Via Host JWT (Silent)

For known hosts with pre-authorized scopes, agent creation is silent — no user interaction needed.

The host signs a JWT with its private key (`sub` = host_id) and sends it with the agent creation request:

```
POST /agent/create
{
    name: "Finance Agent",
    public_key: { ... },
    scopes: ["reports.read"],
    host_jwt: "<signed-jwt>"
}
```

The server:
1. Verifies the host JWT against the stored public key
2. Checks that requested scopes are within the host's pre-authorized budget
3. Creates the agent with `host_id` set
4. Creates `agent_permission` rows for the granted scopes

No user session, no approval screen, no round-trip. This is the fast path for repeat usage.

### 10.3 Scope Exceeds Pre-Authorization

When an agent requests scopes beyond the host's pre-authorized budget:

1. Server creates the agent but with the requested scopes as `pending` permissions
2. Server triggers device authorization flow (Section 2.3)
3. User approves the additional scopes → permissions flip to `active`
4. Host receives confirmation and begins authenticating


## 11. Key Rotation

### 11.1 Agent Key Rotation

An agent's public key can be rotated without creating a new agent:

1. The owner calls the key rotation endpoint with the agent ID and new public key
2. The server replaces the stored public key immediately
3. The old key stops working; the new key takes effect

This enables key rotation without losing the agent's identity, permissions, or history.

### 11.2 Host Key Rotation

For hosts registered with a static public key, key rotation follows the same pattern as agents — the owner provides a new public key via the host update endpoint.

For hosts registered with a JWKS URL (Section 9.3), key rotation is handled by the host itself:

1. The host publishes a new key to its JWKS endpoint (keeping the old key active)
2. The host starts signing new JWTs with the new key
3. After a transition period, the host removes the old key from its JWKS endpoint
4. The server automatically picks up the change on its next JWKS cache refresh

No server-side action is needed. The host manages its own key lifecycle.

## 12. Auditability

Every request in the Agent Auth protocol produces a complete attribution chain:

```
Request → Agent (who) → Permissions (what, granted by whom) → Host (created through) → User (acts for)
```

For any action, the server can answer:
- **Which agent** performed this action? (agent ID, name, keypair)
- **What was it authorized to do?** (active permissions at the time)
- **Who authorized each permission?** (granted_by on each permission row)
- **On what resource?** (reference_id on each permission)
- **Which host created this agent?** (host_id → host record)
- **Which user does this agent act for?** (user_id)

This is fundamentally different from OAuth token sharing, where the answer to "which agent did this?" is "we don't know — some agent using the client's token."

## 13. Security Considerations

### 13.1 Private Key Protection

Agent and host private keys MUST never be transmitted to or stored on the server. The server only stores public keys.

### 13.2 JWT Lifetime

JWTs SHOULD have a maximum age of 60 seconds. Short lifetimes limit the window for token misuse even without request binding.

### 13.3 Request Binding

For sensitive operations, clients SHOULD use request binding (Section 4.3) to cryptographically bind each JWT to the specific HTTP request. This prevents a stolen JWT from being used for unintended requests.

### 13.4 Replay Protection

All JWTs MUST include a unique `jti` claim. Servers MUST maintain a JTI replay cache and reject duplicate `jti` values within the JWT's lifetime window.

### 13.5 Revocation Cascade

When a host is revoked, all agents created under it MUST also be revoked. When an agent is revoked, all its permissions MUST be deleted.

### 13.6 Fresh Session Enforcement

Sensitive operations (approving scope escalations) SHOULD require the user to have authenticated recently. The `fresh_session_window` parameter defines the maximum session age for these operations.

### 13.7 Scope Validation

Servers MAY validate requested scopes against a known set before granting them. Servers SHOULD support a blocked scopes list to prevent specific scopes from ever being granted.
