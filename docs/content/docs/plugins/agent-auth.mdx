The Agent Auth plugin gives AI agent runtimes their own identity so they can act **for** users without impersonating them. Instead of sharing user tokens, each agent gets its own keypair, scopes, and audit trail.

**Core principle:** Agents are tools we hire, not users we pretend to be.

## Features

- [Device authorization connect flow](#connecting-an-agent) ã user approves in browser, no manual tokens
- Ed25519 keypair-based authentication (agent owns its private key)
- [Scoped permissions](#roles-and-scopes) via roles and scopes
- [Agent session resolution](#agent-session-resolution) via a `before` hook
- [Key rotation](#rotate-key) without downtime
- [Revocation](#revoke-an-agent) with credential wipe
- [Token budgets](#token-budgets) ã per-agent, per-user, per-org, and per-workgroup limits
- [Workgroups](#workgroups) ã group agents within an organization with their own budgets
- [Agent SDK](#agent-sdk) for agent runtimes
- [MCP server](#mcp-server) for Cursor, Claude Desktop, and other AI tools
- [Agent Gateway](#agent-gateway) ã proxy third-party MCP servers (GitHub, Slack, etc.) through agent scopes (separate plugin)
- [Dynamic tool discovery](#the-complete-flow) ã agents discover available tools and request only what they need
- [Pluggable storage](#storage-options) ã memory (default), file, or database
- [AAP-compatible JWT claims](#aap-format) (IETF Agent Authorization Profile)

## How It Works

```
1. Agent generates an Ed25519 keypair (private key never leaves the agent)
2. Agent connects via device authorization ã user approves in their browser
3. Agent's public key is registered with the app
4. For each request, the agent signs a short-lived JWT with its private key
5. Better Auth verifies the JWT, resolves the agent session, and attaches it to the request
```

The agent's identity is its keypair. The private key stays on the agent's machine. The server only ever sees the public key.

## Connecting an Agent

The recommended way to connect an agent is via the **device authorization flow** (like `gh auth login` or `gcloud auth login`). The agent displays a code, the user approves in their browser, and the agent is registered. No manual token copying needed.

<Callout>
This requires the `deviceAuthorization()` plugin to be enabled alongside `agentAuth()`.
</Callout>

```ts title="auth.ts"
import { betterAuth } from "better-auth"
import { agentAuth } from "better-auth/plugins"
import { deviceAuthorization } from "better-auth/plugins" // [!code highlight]

export const auth = betterAuth({
    plugins: [
        agentAuth({ /* ... */ }),
        deviceAuthorization(), // [!code highlight]
    ]
})
```

### Via CLI

```bash
npx better-auth agent connect https://myapp.com --name "My Agent" --scopes "reports.read,email.send"
```

This will:
1. Generate a keypair (if one doesn't exist)
2. Display a URL and code for the user to approve
3. Poll until the user approves in their browser
4. Register the agent's public key with the app
5. Store the connection locally

```

  https://myapp.com/device?user_code=ABCD-1234

Or go to https://myapp.com/device and enter code:
  ABCD-1234

Waiting for approval...

  Agent ID: agt_9f3c2e
  Scopes: reports.read, email.send
  App: https://myapp.com
```

### Via SDK (Programmatic)

```ts title="agent-setup.ts"
import { connectAgent } from "better-auth/plugins/agent-auth/agent-client"

const result = await connectAgent({
    appURL: "https://myapp.com",

// result.privateKey ã store this securely
// result.publicKey ã already registered with the app
```

### Why Device Auth?

- **Familiar UX** ã same flow as `gh auth login`, `gcloud auth login`, Spotify on TVs
- **Works from anywhere** ã SSH sessions, remote servers, CI/CD, headless containers
- **Secure** ã user authenticates in their trusted browser, agent never sees passwords
- **Multi-machine** ã each machine generates its own keypair and gets its own agent ID
- **Standard** ã RFC 8628 (OAuth 2.0 Device Authorization Grant)

## Agent Session Resolution

The plugin installs a `before` hook that intercepts every request. When a `Bearer` token containing a JWT is detected:

1. Decodes the JWT to extract the agent ID (`sub` claim)
2. Looks up the agent in the database
3. Verifies the JWT signature against the stored public key
4. Loads the user who owns the agent
5. Attaches an `agentSession` to the request context

### Protecting API Routes

Use `getAgentSession` to check if the request is from an agent:

```ts title="api-route.ts"
export async function GET(req: Request) {
    // Try normal user session first
    const session = await auth.api.getSession({ headers: req.headers })

    // Try agent session
    const agentSession = await auth.api.getAgentSession({
        headers: req.headers,
    })

    if (session) {
        // Regular user request
        const reports = await getReportsForUser(session.user.id)
        return Response.json(reports)
    }

    if (agentSession) {
        // Agent request ã check scopes
        if (!agentSession.agent.scopes.includes("reports.read")) {
            return new Response("Forbidden", { status: 403 })
        }
        const reports = await getReportsForUser(agentSession.user.id)
        return Response.json(reports)
    }

    return new Response("Unauthorized", { status: 401 })
}
```

The `agentSession` object contains:

```ts
{
    agent: {
        id: string
        name: string
        scopes: string[]
        role: string | null
        orgId: string | null
        workgroupId: string | null
        createdAt: Date
        metadata: Record<string, unknown> | null
    },
    user: {
        id: string       // the user who hired this agent
        name: string
        email: string
    }
}
```

## Agent SDK

The Agent SDK is for **agent runtimes** ã the process that actually makes API calls. It handles keypair generation and JWT signing.

```ts title="agent-runtime.ts"
import {
    createAgentClient,
    generateKeypair,
} from "better-auth/plugins/agent-auth/agent-client"

// 1. Generate a keypair (one-time, store the private key securely)
const { publicKey, privateKey } = await generateKeypair()

// 2. Register publicKey with the app (via dashboard or API)
// ... get back agentId ...

// 3. Create an authenticated client
const agent = createAgentClient({
    baseURL: "https://your-app.com",
    agentId: "agt_abc123",
    privateKey, // signs a fresh JWT per request
})

// 4. Make authenticated requests
const res = await agent.fetch("/api/reports/Q4")
const data = await res.json()

// 5. Check own session
const session = await agent.getSession()
// { agent: { id, name, scopes, ... }, user: { id, name, email } }
```

### Credential Storage

The private key should be stored securely and never committed to version control.

| Environment | Recommended storage |
|---|---|
| Local dev | Environment variable or `~/.better-auth/agents/keypair.json` |
| Production | Secret manager (AWS Secrets Manager, Vault, etc.) |
| CI/CD | Pipeline secrets / environment variables |

<Callout type="warn">
Never store private keys in git, logs, or client-side code. The private key is the agent's identity ã if compromised, revoke the agent immediately and rotate the key.
</Callout>

## MCP Server

The Agent Auth plugin includes a standalone MCP server that lets AI tools (Cursor, Claude Desktop, etc.) manage agent connections through natural language. The server exposes tools for agent management: `connect_agent`, `list_agents`, `disconnect_agent`, `agent_status`, and `agent_request`.

### Setup

Add the MCP server to your AI tool's configuration:

```json title=".cursor/mcp.json"
{
    "mcpServers": {
        "better-auth": {
            "command": "npx",
            "args": ["better-auth", "agent"],
            "env": {
                "BETTER_AUTH_URL": "http://localhost:3000"
            }
        }
    }
}
```

<Callout>
The MCP server requires `@modelcontextprotocol/sdk` as a peer dependency. Install it alongside better-auth: `npm install @modelcontextprotocol/sdk`.
</Callout>

### Environment Variables

Only secrets go in the MCP client config. Provider configuration lives in [your plugin config](#mcp-gateway).

| Variable | Description |
|---|---|
| `BETTER_AUTH_URL` | Your app URL (required for provider auto-discovery) |
| `BETTER_AUTH_ENCRYPTION_KEY` | Encrypt agent keypairs at rest (recommended for file storage) |
| `BETTER_AUTH_AGENT_COOKIE` | Optional: session cookie for direct registration (skips device auth) |
| `BETTER_AUTH_AGENT_TOKEN` | Optional: bearer token alternative to cookies |

<Callout>
When no cookie/token is configured, the MCP server uses the **device authorization flow**. The AI tool opens the browser automatically ã the user approves, and the agent is registered. No manual token copying needed.
</Callout>

### MCP Tools

Once configured, the following tools are available to your AI assistant:

| Tool | Description |
|---|---|
| `connect_agent` | Connect to an app. Creates a fresh agent identity (device auth) or reuses an existing one via `agentId`. |
| `list_agents` | List all agent connections in the current session. |
| `agent_status` | Check if a connection is healthy by verifying the agent session. |
| `agent_request` | Make an authenticated API request to a connected app. |
| `disconnect_agent` | Revoke an agent server-side and remove the local connection. |

Each `connect_agent` call creates a **unique agent identity** with its own Ed25519 keypair. The AI passes the returned `agentId` to all subsequent tool calls within the same conversation.

**Example conversation:**

```
You: "Connect to https://myapp.com with reports.read scope"
AI:  ã calls connect_agent(url: "https://myapp.com", name: "Report Agent", scopes: ["reports.read"])
     ã browser opens for approval
     "Approve the connection in your browser."

You: (approves in browser)
AI:  "Connected! Agent ID: abc123. Scopes: reports.read."

You: "Fetch the Q4 report"
AI:  ã calls agent_request(agentId: "abc123", path: "/api/reports/Q4")
     "200 OK ã { ... report data ... }"
```

## Agent Gateway

The Agent Gateway is a **separate plugin** (`agentGateway`) that lets your AI agents connect to **third-party MCP servers** (GitHub, Slack, Google Drive, etc.) through Better Auth's security model. Instead of giving the AI tool direct access to third-party APIs, the gateway acts as a proxy ã agents must be authorized and can only use the tools they were approved for.

<Callout>
The Agent Gateway requires the `agentAuth` plugin to be installed alongside it. It builds on top of agent identity, scopes, and session management.
</Callout>

### How It Works

```
AI Tool (Cursor, Claude Desktop)
    ã stdio
CLI Gateway (npx better-auth agent)
    ããã Agent tools (connect_agent, agent_request, ...)
    ããã Provider tools (github.create_issue, slack.send_message, ...)
            ã stdio / sse
        Third-party MCP servers
```

The gateway:
1. Connects to third-party MCP servers (via stdio or SSE)
2. Discovers all their tools on startup
3. Exposes them as **namespaced tools** (e.g. `github.create_issue`)
4. Enforces **per-agent scope checks** ã agents can only call tools they were approved for

### Provider Management

The gateway exposes REST endpoints for managing providers at runtime. By default, only admin users can register and delete providers.

```ts
// Register a new provider
await fetch("/api/auth/agent/gateway/provider/register", {
    method: "POST",
    headers,
    body: JSON.stringify({
        name: "slack",
        displayName: "Slack",
        transport: "stdio",
        command: "npx",
        args: ["-y", "@anthropic/slack-mcp"],
        orgId: "org_123",         // optional ã scope to an org
        workgroupId: "wg_456",    // optional ã scope to a workgroup
    }),

// List providers (with optional org/workgroup filter)

    method: "POST",
    headers,
    body: JSON.stringify({ name: "slack" }),
```

Configure who can manage providers via `authorizeProviderManagement`:

```ts
agentGateway({
    authorizeProviderManagement: async (user) => {
        return user.role === "admin" || user.role === "org_owner"
    },

```

### The Complete Flow

Here's what happens when a user asks their AI agent to do something that requires a third-party tool:

```
1. User: "Create a GitHub issue about the login bug"

2. Agent calls list_gateway_tools ã discovers available tools:
   ## GitHub (28 tools)
     - github.create_issue: Create a new issue
     - github.search_repositories: Search repositories
     - github.create_pull_request: Create a pull request
     ...

3. Agent picks what it needs and calls connect_agent:
   connect_agent(
     url: "http://localhost:3000",
     name: "Issue Creator Agent",
     scopes: ["github.create_issue"]  ã only what's needed
   )

4. Browser opens ã consent screen shows:
   "Issue Creator Agent wants access to:
    ã GitHub ã create_issue: Create a new issue"
   ã User approves

5. Agent calls github.create_issue(agentId: "abc123", ...)
   ã Works ã

6. Agent tries github.delete_repository(agentId: "abc123", ...)
   ã "Scope denied" ã
```

The key principle: **agents request only the tools they need, and users see exactly what they're approving.**

### Consent Screen (Rich Authorization Requests)

When an agent requests access, the `deviceVerify` endpoint returns structured `authorization_details` following [RFC 9396 (Rich Authorization Requests)](https://datatracker.ietf.org/doc/html/rfc9396). This gives your consent page fine-grained, human-readable information about what the agent wants:

```ts title="consent-page.tsx"
// GET /api/auth/device?user_code=ABCD-1234
const response = await fetch(`/api/auth/device?user_code=${userCode}`)
const data = await response.json()

// data.client_name ã "Issue Creator Agent"
// data.scope ã "github.create_issue github.search_repositories"
// data.authorization_details ã [
//   {
//     type: "mcp_tool",
//     locations: ["GitHub"],
//     actions: ["execute"],
//     identifier: "create_issue",
//     description: "Create a new issue in a GitHub repository"
//   },
//   {
//     type: "mcp_tool",
//     locations: ["GitHub"],
//     actions: ["execute"],
//     identifier: "search_repositories",
//     description: "Search for GitHub repositories"
//   }
// ]
```

The `authorization_details` array uses RFC 9396 common fields:

| Field | Description |
|---|---|
| `type` | Authorization type ã `"mcp_tool"` for gateway tools |
| `locations` | Provider name or resource URL (e.g. `["GitHub"]`) |
| `actions` | What the agent will do (e.g. `["execute"]`) |
| `identifier` | Specific tool name (e.g. `"create_issue"`) |
| `description` | Human-readable description of the tool |

The same `authorization_details` is also returned in the token response (per RFC 9396 ôÏ7), so the agent knows exactly what was granted.

```ts title="my-mcp-server.ts"
for (const tool of tools) {
    mcpServer.tool(tool.name, tool.description, tool.inputSchema, tool.handler)
}
```

### Storage Options

Each agent connection stores the agent's keypair, scopes, and app URL. You choose where to persist this data based on your use case.

#### Memory Storage (Default)

Connections live in-process memory. When the MCP server process exits, all connections are gone and the AI must reconnect (requiring user approval). Best for interactive AI tools like Cursor and Claude Desktop where each session should have its own agent identity.

```ts
import { createMemoryStorage } from "better-auth/plugins/agent-auth/mcp-storage-memory"

const tools = createAgentMCPTools({
    storage: createMemoryStorage(),
})
```

#### File Storage

Connections persist to disk as JSON files. Survives process restarts. Best for long-running agent scripts or local development where you don't want to re-approve on every restart.

```ts
import { createFileStorage } from "better-auth/plugins/agent-auth/mcp-storage-fs"

const tools = createAgentMCPTools({
    storage: createFileStorage({
        directory: "~/.better-auth/agents",
        encryptionKey: process.env.BETTER_AUTH_ENCRYPTION_KEY, // optional
    }),
})
```

When `encryptionKey` is provided, agent keypairs are encrypted at rest using XChaCha20-Poly1305.

Files are stored at `~/.better-auth/agents/`:

```
~/.better-auth/agents/
ããã connections/
ã   ããã <agentId>.json   # One file per agent (keypair + connection data)
ããã pending-flows.json   # In-progress device auth flows
```

All secret files are written with `chmod 0600` (owner-only).

#### Database Storage

For production agent deployments or when you need to manage many agents, implement the `MCPAgentStorage` interface with your own database:

```ts title="my-db-storage.ts"
import type { MCPAgentStorage } from "better-auth/plugins/agent-auth/mcp-tools"

const dbStorage: MCPAgentStorage = {
    async getConnection(agentId) {
        const row = await db.query("SELECT * FROM agent_connections WHERE agent_id = ?", [agentId])
        if (!row) return null
        return {
            appUrl: row.app_url,
            keypair: JSON.parse(row.keypair),
            name: row.name,
            scopes: JSON.parse(row.scopes),
        }
    },
    async saveConnection(agentId, connection) {
        await db.query(
            "INSERT INTO agent_connections (agent_id, app_url, keypair, name, scopes) VALUES (?, ?, ?, ?, ?) ON CONFLICT (agent_id) DO UPDATE SET ...",
            [agentId, connection.appUrl, JSON.stringify(connection.keypair), connection.name, JSON.stringify(connection.scopes)]
        )
    },
    async removeConnection(agentId) {
        await db.query("DELETE FROM agent_connections WHERE agent_id = ?", [agentId])
    },
    async listConnections() {
        const rows = await db.query("SELECT agent_id, app_url, name, scopes FROM agent_connections")
        return rows.map(r => ({
            agentId: r.agent_id,
            appUrl: r.app_url,
            name: r.name,
            scopes: JSON.parse(r.scopes),
        }))
    },
}

const tools = createAgentMCPTools({ storage: dbStorage })
```

<Callout type="warn">
The `keypair` field contains the agent's **private key**. If you store it in a database, encrypt it at rest (AES-256 or similar) and restrict access to the MCP service account only. Never expose private keys in API responses or logs.
</Callout>

### Agent Lifecycle

The gateway handles cleanup automatically:

- **Process shutdown** (SIGINT, SIGTERM) ã revokes all active agents server-side before exiting
- **Transport close** (stdio pipe breaks) ã same cleanup as shutdown
- **Server-side TTL** ã for crash recovery, implement `expiresAt` on agents and run periodic cleanup

<Callout>
Each conversation in Cursor/Claude Desktop shares a single MCP server process. The agent uses a unique `agentId` per conversation to maintain separate identities. Memory storage (the default) ensures a clean slate on every MCP restart.
</Callout>

## Token Budgets

Agent Auth tracks token consumption per-agent (input and output tokens). You can set budgets at multiple levels to control spending:

```ts title="auth.ts"
agentAuth({
    maxTokensPerAgent: 100_000,       // per individual agent
    maxTokensPerUser: 500_000,        // across all of a user's agents
    maxTokensPerOrg: 1_000_000,       // across all agents in an organization
    maxTokensPerWorkgroup: 250_000,   // across all agents in a workgroup
})
```

When an agent reports activity via `POST /agent/log-activity` with `inputTokens` and `outputTokens`, the plugin checks all applicable budgets. If any budget would be exceeded, the request is rejected with a `403 FORBIDDEN` error.

| Budget | Scope | Error Code |
|---|---|---|
| `maxTokensPerAgent` | Single agent | `TOKEN_BUDGET_EXCEEDED` |
| `maxTokensPerUser` | All agents owned by a user | `USER_TOKEN_BUDGET_EXCEEDED` |
| `maxTokensPerOrg` | All agents in an organization | `ORG_TOKEN_BUDGET_EXCEEDED` |
| `maxTokensPerWorkgroup` | All agents in a workgroup | `WORKGROUP_TOKEN_BUDGET_EXCEEDED` |

Set any value to `0` (the default) to disable that budget level.

### Token Usage API

```ts
// Get token usage for an agent
const { data } = await authClient.agent.getTokenUsage({
    query: { agentId: "agent-id" }
}, { headers })
// { agentTokens: { input, output, total }, userTokens: { input, output, total } }
```

## Workgroups

Workgroups let you **group agents within an organization**. This is useful for enterprise setups where you want to organize agents by team, project, or department ã each with its own token budget.

### Create a Workgroup

```ts
await fetch("/api/auth/agent/workgroup/create", {
    method: "POST",
    headers,
    body: JSON.stringify({
        name: "Finance Team",
        orgId: "org_123",
        description: "Agents for the finance department",
        maxTokens: 500_000,  // optional per-workgroup budget (0 = unlimited)
    }),
})
```

### List Workgroups

```ts
const res = await fetch("/api/auth/agent/workgroup/list?orgId=org_123", { headers })
const { workgroups } = await res.json()
// [{ id, name, orgId, description, maxTokens, createdAt, updatedAt }]
```

### Update a Workgroup

```ts
await fetch("/api/auth/agent/workgroup/update", {
    method: "POST",
    headers,
    body: JSON.stringify({
        workgroupId: "wg_456",
        name: "Finance & Accounting",
        maxTokens: 750_000,
    }),
})
```

### Delete a Workgroup

```ts
await fetch("/api/auth/agent/workgroup/delete", {
    method: "POST",
    headers,
    body: JSON.stringify({ workgroupId: "wg_456" }),
})
```

When a workgroup is deleted, agents in that workgroup have their `workgroupId` set to `null` (they remain active but unassigned).

### Assigning Agents to Workgroups

Pass `workgroupId` when creating an agent:

```ts
await authClient.agent.create({
    name: "Budget Analyzer",
    publicKey: agentPublicKey,
    scopes: ["reports.read"],
    orgId: "org_123",
    workgroupId: "wg_456",
}, { headers })
```

Filter agents by workgroup:

```ts
const { data } = await authClient.agent.list({
    query: { orgId: "org_123", workgroupId: "wg_456" }
}, { headers })
```

## Roles and Scopes

You can define named roles that map to a set of scopes:

```ts title="auth.ts"
agentAuth({
    roles: {
        reader: ["reports.read"],
        writer: ["reports.read", "reports.write", "email.send"],
        finance: ["reports.read", "invoices.read", "invoices.write"],
    },
    defaultRole: "reader",
})
```

When creating an agent with a `role`, the scopes are automatically resolved from the role config. You can also pass explicit `scopes` to override.

## AAP Format

The plugin supports [IETF Agent Authorization Profile](https://datatracker.ietf.org/doc/draft-ietf-oauth-agent-authorization/) compatible JWT claims. Set `jwtFormat: "aap"` to enable structured claims:

```ts title="auth.ts"
agentAuth({
    jwtFormat: "aap", // [!code highlight]
})
```

This adds AAP-standard claims like `aap_agent` to the JWT payload alongside the standard `sub`, `iat`, `exp` claims.
