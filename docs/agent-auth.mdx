`Ed25519` `Hosts` `Permissions` `MCP`

The Agent Auth plugin gives AI agents their own cryptographic identity so they can act **for** users without sharing or impersonating user tokens. Each agent gets its own Ed25519 keypair, and permissions are granted per-agent by individual users — with support for multi-grantor authorization, resource scoping, and request-bound proof-of-possession.

**Core principle:** Agents are tools we hire, not users we pretend to be.

## Features

- Ed25519 keypair-based authentication (agent owns its private key)
- [Hosts](#hosts) — persistent app-level consent for silent agent creation
- [Three-state lifecycle](#three-state-lifecycle) — `active` → `expired` → `revoked` with proof-of-possession reactivation
- [Transparent reactivation](#transparent-reactivation) — expired agents auto-reactivate on valid JWT
- [Three lifetime clocks](#lifetime-clocks) — sliding TTL, max lifetime, absolute lifetime
- [Permission-based authorization](#permissions-and-scopes) via roles, wildcard matching, and blocked scopes
- [Scope escalation](#scope-escalation) with fresh session enforcement
- [Request binding](#request-binding) — optional DPoP-style JWT claims for method, path, and body hash
- [Discovery endpoint](#discovery) for capability negotiation
- [JTI replay detection](#jti-replay-detection) for agent and host JWTs
- [Device authorization connect flow](#connecting-an-agent) — user approves in browser, no manual tokens
- [Key rotation](#rotate-key) without downtime
- [Revocation](#revoke-an-agent) with credential wipe and cascade
- [Agent SDK](#agent-sdk) for agent runtimes
- [MCP server](#mcp-server) for Cursor, Claude Desktop, and other AI tools
- [Pluggable storage](#storage-options) — memory (default), file, or database
- [AAP-compatible JWT claims](#aap-format) (optional structured claim format)

## How It Works

Agent Auth follows an **Agent → Host → Server** architecture:

- **Agent** — an AI identity with its own Ed25519 keypair
- **Host** — a registered client runtime (Cursor, MCP server, CLI) with pre-authorized scope budget
- **Server** — your app's auth backend running the Agent Auth plugin

The flow:

1. **User registers a host** — a persistent app-level consent record with a scope budget and its own keypair
2. **Agent generates a keypair** — an Ed25519 keypair is generated client-side; the private key never leaves the agent
3. **Agent connects** — via device authorization (user approves in browser) or host JWT (silent)
4. **Public key is registered** — the server stores the agent's public key and creates permission rows from the host's scope budget
5. **Agent signs JWTs** — for each request, the agent signs a short-lived JWT with its private key
6. **Server verifies** — Better Auth verifies the JWT, resolves the agent session with active permissions, and attaches it to the request
7. **Auto-reactivation** — if the agent has expired, the server transparently reactivates it (no extra round-trip)

## Hosts

Hosts are persistent, app-level consent records that represent a **registered client runtime** (Cursor, MCP server, CLI tool). They enable **silent agent creation** — agents can be registered without requiring a user session at the time of creation.

Each host has its own Ed25519 keypair (matching the agent model — no bearer tokens). When an agent is created via a host JWT, it inherits the host's `scopes` as its initial permissions.

<Callout type="info">
If a `kid` in the public key matches an existing host for the same user, the existing host is reactivated and updated instead of creating a duplicate (idempotent creation). If the existing host is revoked, the request is rejected.
</Callout>

## Three-State Lifecycle

Both agents and hosts follow a three-state lifecycle:

```
      ┌────────────── proof-of-possession ──────────────┐
      │                                                 │
      ▼              TTL / maxLifetime                  │
  ┌────────┐  ─────────────────────────────►  ┌─────────┴─┐
  │ active │                                  │  expired   │
  └────────┘  ◄─────────────────────────────  └───────────┘
      │         (permissions decay to host)         │
      │                                             │
      │   absoluteLifetime / explicit revoke        │  absoluteLifetime / explicit revoke
      ▼                                             ▼
  ┌─────────┐                                 ┌─────────┐
  │ revoked │                                 │ revoked │
  └─────────┘                                 └─────────┘
   (permanent, key wiped)                      (permanent, key wiped)
```

- **Active** — the entity is operational; each request extends the sliding TTL
- **Expired** — the TTL or max lifetime has elapsed; the public key is retained; reactivatable via proof-of-possession
- **Revoked** — permanent; public key is wiped; cannot be reactivated

### Transparent Reactivation

When the middleware detects a request from an expired agent with a valid JWT signature, it **automatically reactivates** the agent — no separate API call needed. The response proceeds normally as if the agent were active.

On reactivation, permissions decay to the host's `scopes` (if a host exists). If no host is associated, existing permissions are kept as-is. The `activatedAt` and `expiresAt` timestamps are reset.

### Lifetime Clocks

Three independent clocks govern agent and host lifetimes:

| Clock | Measured From | Resets On Reactivation | Effect When Elapsed |
|---|---|---|---|
| `agentSessionTTL` | last request (`expiresAt`) | Yes | → `expired` (reactivatable) |
| `agentMaxLifetime` | `activatedAt` | Yes (resets `activatedAt`) | → `expired` (reactivatable) |
| `absoluteLifetime` | `createdAt` | **Never** | → `revoked` (permanent) |

## Agent Session Resolution

The plugin installs a `before` hook that intercepts every request. When a `Bearer` token containing a JWT is detected:

1. Decodes the JWT to extract the agent ID (`sub` claim)
2. Looks up the agent in the database
3. Checks the three-state lifecycle (revoked → reject, expired → attempt transparent reactivation)
4. Verifies the JWT signature against the stored public key
5. Checks for JTI replay
6. Verifies [request binding](#request-binding) claims if present (`htm`, `htu`, `ath`)
7. Loads the user who owns the agent
8. Loads active permissions from the `agentPermission` table
9. Attaches an `agentSession` to the request context
10. Updates `lastUsedAt` and extends `expiresAt` (heartbeat)

### Protecting API Routes

```ts title="api-route.ts"
export async function GET(req: Request) {
    const session = await auth.api.getSession({ headers: req.headers })
    const agentSession = await auth.api.getAgentSession({ headers: req.headers })

    if (session) {
        const reports = await getReportsForUser(session.user.id)
        return Response.json(reports)
    }

    if (agentSession) {
        const hasScope = agentSession.agent.permissions.some(
            (p) => p.scope === "reports.read"
        )
        if (!hasScope) {
            return new Response("Forbidden", { status: 403 })
        }
        const reports = await getReportsForUser(agentSession.user.id)
        return Response.json(reports)
    }

    return new Response("Unauthorized", { status: 401 })
}
```

The `agentSession` object:

```ts
type AgentSession = {
    agent: {
        id: string
        name: string
        permissions: Array<{
            scope: string
            referenceId: string | null
            grantedBy: string
            status: string
        }>
        hostId: string
        createdAt: Date
        activatedAt: Date | null
        metadata: Record<string, unknown> | null
    }
    user: {
        id: string
        name: string
        email: string
    }
}
```

## Permissions and Scopes

Authorization is managed through the `agentPermission` table. Each row represents a single scope grant, supporting multi-grantor permissions where different users can grant different scopes to the same agent.

### Roles

```ts title="auth.ts"
agentAuth({
    roles: {
        reader: ["reports.read"],
        writer: ["reports.read", "reports.write", "email.send"],
        finance: ["reports.read", "invoices.read", "invoices.write"],
    },
    defaultRole: "reader",
})
```

When creating an agent with a `role`, scopes are automatically resolved from the role config and stored as individual permission rows.

### Wildcard Matching

Scopes support hierarchical wildcard matching:

- `"github.*"` covers `"github.create_issue"`, `"github.list_repos"`, etc.
- `"*"` covers everything

Wildcards are respected across scope checks, subset validation, and scope merging.

### Blocked Scopes

Prevent specific scopes from ever being granted:

```ts title="auth.ts"
agentAuth({
    blockedScopes: ["admin.delete", "system.*"], // [!code highlight]
})
```

Blocked scopes are enforced on agent creation, host creation, and scope escalation approval.

### Scope Escalation

## Request Binding

Agent JWTs support optional DPoP-style request binding. When present, the server verifies that the JWT was intended for the specific request being made:

| Claim | Description |
|---|---|
| `htm` | HTTP method (e.g. `"GET"`, `"POST"`) |
| `htu` | HTTP path (e.g. `"/api/reports"`) |
| `ath` | SHA-256 hash of the request body (Base64url-encoded) |

If any of these claims are present in the JWT, the server verifies them against the actual request. A mismatch results in `REQUEST_BINDING_MISMATCH`.

The Agent SDK automatically includes request binding claims when using `createAgentClient`:

```ts title="agent-runtime.ts"
const agent = createAgentClient({
    baseURL: "https://your-app.com",
    agentId: "agt_abc123",
    privateKey,
})

// Every request automatically includes htm, htu, and ath claims
const res = await agent.fetch("/api/reports/Q4")
```

## JTI Replay Detection

All proof-of-possession JWTs (agent requests, host JWTs, and reactivation proofs) are checked against an in-memory JTI cache. If a JWT's `jti` claim has been seen within its `jwtMaxAge` window, the request is rejected with `JWT_REPLAY`.

## Options

`roles` <span className="opacity-70">`Record<string, string[]>`</span>

Named roles mapping to scope arrays. When creating an agent with a `role`, scopes are automatically resolved and stored as permission rows.

`defaultRole` <span className="opacity-70">`string`</span>

Default role assigned to new agents when none is specified.

`allowedKeyAlgorithms` <span className="opacity-70">`string[]`</span>

Allowed key algorithms for agent keypairs. Use JWK curve names (`"Ed25519"`, not `"EdDSA"`). Default: `["Ed25519"]`.

`jwtFormat` <span className="opacity-70">`"simple" | "aap"`</span>

JWT claim format. `"simple"` uses flat claims. `"aap"` uses structured claims compatible with the AAP draft. Default: `"simple"`.

`jwtMaxAge` <span className="opacity-70">`number`</span>

Maximum age for agent JWTs in seconds. Default: `60`.

`agentSessionTTL` <span className="opacity-70">`number`</span>

Sliding TTL for agent and host sessions in seconds. Each authenticated request extends the deadline. Set to `0` to disable. Default: `3600` (1 hour).

`agentMaxLifetime` <span className="opacity-70">`number`</span>

Maximum lifetime from `activatedAt` in seconds. Resets on each reactivation. When elapsed, the entity transitions to `expired` (reactivatable). Set to `0` to disable. Default: `86400` (24 hours).

`absoluteLifetime` <span className="opacity-70">`number`</span>

Absolute lifetime from `createdAt` in seconds. Never resets. When elapsed, the entity is **revoked** (permanent — cannot be reactivated). Set to `0` to disable. Default: `0` (disabled).

`freshSessionWindow` <span className="opacity-70">`number`</span>

Seconds within which the user must have authenticated for sensitive operations (approve-scope). If the session is older, re-authentication is required. Set to `0` to disable. Default: `300` (5 minutes).

`blockedScopes` <span className="opacity-70">`string[]`</span>

Scopes that are always blocked from being granted or escalated. Supports wildcard patterns. Default: `[]`.

`maxAgentsPerUser` <span className="opacity-70">`number`</span>

Maximum number of active agents a single user can have. Set to `0` for unlimited. Default: `25`.

`validateScopes` <span className="opacity-70">`boolean | ((scopes: string[]) => boolean | Promise<boolean>)`</span>

Validate that requested scopes exist before granting them. When `true`, scopes are checked against the union of all role scopes. When a function, receives the scopes array and should return `true` if valid. Default: `false`.

`rateLimit` <span className="opacity-70">`{ window?: number; max?: number; createMax?: number; sensitiveMax?: number } | false`</span>

Rate limiting configuration for agent endpoints. Set to `false` to disable entirely. Default: `{ window: 60, max: 60, createMax: 10, sensitiveMax: 5 }`.

`schema` <span className="opacity-70">`object`</span>

Custom schema overrides for the agent tables.

## Error Codes

| Code | Message |
|---|---|
| `AGENT_NOT_FOUND` | Agent not found. |
| `AGENT_REVOKED` | Agent has been revoked. |
| `AGENT_EXPIRED` | Agent session has expired. |
| `INVALID_JWT` | Invalid or expired agent JWT. |
| `JWT_REPLAY` | JWT has already been used (replay detected). |
| `SCOPE_DENIED` | Agent does not have the required scope. |
| `SCOPE_BLOCKED` | One or more requested scopes are blocked. |
| `UNAUTHORIZED_SESSION` | Unauthorized or invalid session. |
| `FRESH_SESSION_REQUIRED` | A fresh session is required. Please re-authenticate. |
| `INVALID_PUBLIC_KEY` | Invalid public key format. |
| `AGENT_NAME_REQUIRED` | Agent name is required. |
| `INVALID_SCOPES` | Scopes must be an array of strings. |
| `UNKNOWN_SCOPES` | One or more requested scopes are not recognized. |
| `AGENT_LIMIT_REACHED` | Maximum number of active agents reached. |
| `SCOPE_REQUEST_NOT_FOUND` | No pending permission requests found for this agent. |
| `SCOPE_REQUEST_OWNER_MISMATCH` | This agent belongs to a different user. |
| `SCOPE_REQUEST_ALREADY_RESOLVED` | All permission requests have already been resolved. |
| `NO_SCOPE_CHANGES` | No new scopes requested. |
| `HOST_NOT_FOUND` | Agent host not found. |
| `HOST_REVOKED` | Agent host has been revoked. |
| `HOST_EXPIRED` | Agent host has expired. Reactivate it via proof-of-possession. |
| `HOST_REQUIRED` | An active agent host is required to create agents. |
| `REQUEST_BINDING_MISMATCH` | JWT request binding does not match the actual request. |

Error codes are exported from both the server and client packages:

```ts
import { AGENT_AUTH_ERROR_CODES } from "@better-auth/agent-auth"
// or
import { AGENT_AUTH_ERROR_CODES } from "@better-auth/agent-auth/client"
```
